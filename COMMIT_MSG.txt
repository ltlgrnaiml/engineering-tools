fix: Remove /v1 API versioning - enforce contract-level versioning only

## Summary
Removed all `/v1` URL prefixes from API routes across backend and frontend.
Per ADR-0009 and ADR-0017, API versioning is enforced at the contract module 
level (__version__ in Pydantic models), not at the URL level. This fixes 3 out 
of 4 reported runtime bugs (DAT, Pipelines, SOV).

## Bugs Fixed

### ✅ DAT: Cannot Create New Run
- **Root Cause**: Frontend calling `/api/dat/runs`, backend expecting `/api/dat/v1/runs`
- **Fix**: Removed `prefix="/v1"` from DAT APIRouter
- **File**: `apps/data_aggregator/backend/src/dat_aggregation/api/routes.py`

### ✅ Pipelines: Failed to Load
- **Root Cause**: Frontend calling `/api/pipelines`, backend at `/api/v1/pipelines`
- **Fix**: Removed `/v1` from gateway pipeline router
- **Files**: 
  - `gateway/main.py`
  - `apps/homepage/frontend/src/hooks/usePipelines.ts`
  - `apps/homepage/frontend/src/pages/PipelinesPage.tsx`
  - `apps/homepage/frontend/src/pages/PipelineDetailsPage.tsx`

### ✅ SOV: Loading Spinner (Incorrect API Paths)
- **Root Cause**: Frontend calling `/api/sov/v1/analysis` and `/api/datasets/v1/`
- **Fix**: Removed `/v1` from SOV and datasets frontend calls
- **Files**:
  - `apps/sov_analyzer/frontend/src/components/AnalysisConfig.tsx`
  - `apps/sov_analyzer/frontend/src/components/ResultsPanel.tsx`

### ⚠️ PPTX: Validation Blocking Generation (NOT A BUG)
- **Status**: This is **correct behavior** per ADR-0019 workflow enforcement
- **Error**: "Generation blocked per ADR-0019: Generation requires validation to pass"
- **Explanation**: WorkflowFSM correctly requires Four Green Bars validation before generation
- **No Fix Needed**: Frontend must complete validation step before generation attempt

## Architecture Alignment

Per **ADR-0009 (Type-Safety & Contract Discipline)** and **ADR-0017 (Cross-Cutting Guardrails)**:

### Versioning Strategy
- ✅ **Contract-Level Versioning**: Each Pydantic model has `__version__` attribute
- ✅ **No URL Versioning**: API routes are stable (no /v1, /v2 prefixes)
- ✅ **Breaking Changes**: Require version bump in contract module + ADR update
- ✅ **Non-Breaking Changes**: Adding optional fields can be made in-place
- ✅ **CI Enforcement**: `ci/steps/step_openapi_client_drift.ps1` detects breaking changes

### Why No URL Versioning?
1. **Simpler URLs**: Cleaner API paths without /v1 clutter
2. **Contract Discipline**: Versioning at source (Pydantic models), not transport (URLs)
3. **Backward Compatibility**: Breaking changes caught by CI, not hidden in URLs
4. **Solo-Dev Context**: Easier to maintain single API version than multiple URL prefixes
